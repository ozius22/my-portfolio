01 — The Concept

What is Git Merge?

Git merge is the process of integrating changes from one branch into another — bringing your teammates' work together into a single unified codebase.

1

Combine Colleagues' Code

When two or more developers work on different branches, merging brings all their changes together — like assembling puzzle pieces from different teammates.

Alice's code Bob's code → Merged
2

Each Feature, One Merge

Each feature lives on its own branch. Once it's complete and tested, it gets merged. This keeps work isolated, reviewable, and safe.

feature/login → done ✓

feature/dashboard → in review

feature/payments → in progress

3

Protected Branches

main and dev are protected — you cannot push directly. All changes must go through a Pull Request and be approved first.

🔒 main 🔒 dev

03 — The Workflow

The Merge Workflow

From writing code to landing it in main — here's the typical flow every team follows.

1

Create a Feature Branch

git checkout -b

Branch off from dev. Give it a clear descriptive name. Never work directly on main or dev.

$ git checkout -b feature/user-auth
2

Commit Your Changes

git commit

Make atomic, meaningful commits. Each commit should do one thing and have a clear message.

$ git add .

$ git commit -m "feat: add JWT authentication"

3

Open a Pull Request

GitHub / GitLab

Push your branch and open a PR targeting dev. Your colleagues review your code before it's merged.

⬤ feat: add JWT authentication Open

feature/user-auth → dev · 3 commits · +148 -12 lines

👀 2 reviewers 💬 4 comments
4

Merge After Approval

git merge

Once reviewers approve, the PR is merged into dev. When the sprint ends, dev merges into main for release.

$ git checkout dev

$ git merge feature/user-auth

# ✅ Merge made by the 'ort' strategy.

04 — Protected Branches

main & dev are Sacred

These branches are the backbone of your project. They are locked down for a reason — no direct pushes, no shortcuts.

🔒 main PRODUCTION
  • ▸ Live code — what users see in production
  • ▸ Only receives merges from dev
  • ▸ Requires at least 1–2 approvals before merge
  • ▸ CI/CD pipelines must pass (tests, linting)
  • ▸ Direct push → ❌ Rejected by the repository
$ git push origin main
remote: error: GH006: Protected branch update failed.
remote: Required status checks are not passing.
🔒 dev STAGING
  • ▸ Integration branch — features come here first
  • ▸ Receives merges from all feature/* branches
  • ▸ Team tests here before promoting to main
  • ▸ Requires at least 1 approval before merge
  • ▸ All feature branches are based off of this
$ git push origin feature/auth
# ✅ Branch pushed, open a PR → dev

05 — Quick Reference

Git Merge Cheatsheet

Branch off from dev

git checkout -b feature/name

Stage all changes

git add .

Commit with message

git commit -m "feat: thing"

Push your branch

git push origin feature/name

Switch to dev to merge

git checkout dev

Merge feature into dev

git merge feature/name

Pull latest before merging

git pull origin dev

Delete merged branch

git branch -d feature/name

View all branches

git branch -a
