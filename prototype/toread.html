Introduction

What is Nuxt.js?

Nuxt is a full-stack framework built on top of Vue.js. Think of Vue as the engine, and Nuxt as the fully assembled car — it comes with the routing, the folder structure, the SSR support, and the tooling all pre-configured.

When you build a web application with plain Vue, you have to manually set up routing, decide how to structure your files, configure how data is fetched on the server vs. the client, and wire everything together. Nuxt makes these decisions for you with sensible, opinionated defaults — so your team can focus on building features instead of boilerplate.

🏗️ Analogy for students

Vue is like buying all the raw lumber and nails to build a house. Nuxt is like buying a prefabricated house kit — the walls, roof, and foundation are already cut and labeled. You still build the house, but you're not starting from raw materials.

Why do we use Nuxt in this project?

We use Nuxt because it gives us three things out of the box that our Laravel API backend needs:

  1. File-based routing — pages are just files inside the pages/ folder. No manual routing config.
  2. Auto-imports — components and composables are automatically available everywhere. No import statements needed.
  3. Universal $fetch — a built-in HTTP client that works on both the server and client side seamlessly.

Components
Chapter 1

Components — Reusable UI Blocks

A component is a self-contained piece of UI — its own HTML structure, its own logic, and its own style — that can be used anywhere in your app, as many times as you need.

What is a Component?

Imagine you're building a website and you need a "card" — a box with a title, description, and a button — in ten different places. Without components, you'd copy and paste that HTML ten times. If you ever need to change the design, you'd have to update ten different files.

With components, you write that card once in its own file, and then use <MyCard /> wherever you need it. Update the component once, and every place it's used updates automatically.

🍪 Analogy for students

A component is like a cookie cutter. You design the shape once, and you can stamp it out as many times as you want — each cookie (instance) is separate, but they all come from the same mold. You can even give each cookie different toppings (props).

Creating a Component in Nuxt

In Nuxt, you create a component by adding a .vue file inside the components/ folder. That's it. No registration, no import statements needed elsewhere — Nuxt handles that automatically.

Every component file has three parts:

🧩
The Three Sections of a .vue File

<template> — the HTML structure (what the user sees)
<script setup> — the JavaScript logic (how it behaves)
<style> — the CSS styling (optional, scoped to this component)

components/ui/BaseCard.vue — a complete example

<!-- ① TEMPLATE: what the user sees -->

<template>

  <div class="card">

    <h3>{{ title }}</h3> <!-- "title" comes from props -->

    <p>{{ description }}</p>

    <slot /> <!-- whatever is placed between <BaseCard>...</BaseCard> appears here -->

  </div>

</template>


<!-- ② SCRIPT: the logic -->

<script setup lang="ts">

// defineProps: declare what data this component accepts from its parent

const props = defineProps<{

  title: string // required — parent MUST pass this

  description?: string // optional — the ? means "not required"

}>()

</script>

Auto-Import & Naming Convention

This is one of Nuxt's most powerful features. You never need to write an import statement for your components. Nuxt scans the components/ folder automatically and makes every component globally available.

The component name is derived from its file path:

File Location Used in template as Why?
components/AppHeader.vue <AppHeader /> Root-level, name as-is
components/ui/BaseButton.vue <UiBaseButton /> Folder name becomes prefix
components/forms/SearchInput.vue <FormsSearchInput /> Folder name becomes prefix
💡
Teaching Tip — Why the prefix?

The folder prefix prevents naming collisions. If you have ui/Button.vue and forms/Button.vue, they become <UiButton /> and <FormsButton /> — no conflict.

Props — Sending Data INTO a Component

Props are how a parent component passes information to a child component. Think of props as the parameters of a function — when you call the function (use the component), you pass values in.

You define what props a component accepts using defineProps<T>():

Props — definition & usage

// In the component (BaseButton.vue) — DEFINING props:

const props = defineProps<{

  label: string // required string

  disabled?: boolean // optional boolean (? = optional)

  variant: 'primary' | 'ghost' // must be one of these two values

}>()


// In a page or parent component — USING the component:

<BaseButton

  label="Save Changes" <!-- passing a static string -->

  :disabled="isLoading" <!-- : means it's dynamic (a variable) -->

  variant="primary"

/>

📌
Static vs Dynamic Props

Without a colon (label="text") — the value is a plain string.
With a colon (:disabled="isLoading") — the value is JavaScript (a variable, expression, or computed).
This is called v-bind, and the colon is its shorthand.

Emits — Sending Events OUT of a Component

Props go into a component. Emits go out. When something happens inside a component (user clicks a button, submits a form), the component emits an event that the parent can listen to and respond to.

🔔 Analogy for students

Think of a doorbell. The button (child component) doesn't know what will happen when pressed — it just rings (emits). The house owner (parent) decides what to do when they hear the ring. The child only does one thing: announce something happened.

Emits — definition & usage

// In the component — DEFINING emits:

const emit = defineEmits(['submit', 'cancel'])


// Inside a function, FIRE the event:

const handleClick = () => {

  emit('submit', formData) // 'submit' is event name, formData is the payload

}


// In the PARENT — LISTENING for the event:

<LoginForm @submit="handleFormSubmit" />

// ^ @submit is short for v-on:submit

// ^ handleFormSubmit is a function in the parent that receives formData

Slots — Flexible Content Inside a Component

Sometimes you want a component to be a container — it provides the shell (border, padding, header) but lets the parent decide what goes inside. That's what <slot /> is for.

Slots — definition & usage

// BaseModal.vue — defines a slot:

<template>

  <div class="modal-backdrop">

    <div class="modal-box">

      <slot /> <!-- parent content goes here -->

    </div>

  </div>

</template>


// Parent page — uses BaseModal and puts content in the slot:

<BaseModal>

  <h2>Confirm Delete</h2>

  <p>Are you sure? This cannot be undone.</p>

  <BaseButton label="Delete" variant="danger" />

</BaseModal>

Common Student Questions — Components

Do I need to import a component before using it?
No! Nuxt automatically imports everything from your components/ folder. This is one of Nuxt's biggest conveniences. Just create the file and use it.
When should I create a new component?
A good rule of thumb: if the same UI appears more than once, or if a piece of UI has its own logical purpose (a card, a modal, a form field), make it a component. Also extract it when a page file gets too long — split it into smaller components.
What's the difference between props and data (ref)?
ref() or reactive() is data the component owns internally — it manages it itself. Props are data that comes from the outside — from the parent. A component should never modify its own props directly.

Chapter 1 Summary — Components

  • A component is a reusable, self-contained piece of UI in a .vue file.
  • Nuxt auto-imports everything from components/ — no manual imports needed.
  • Folders create naming prefixes: ui/Card.vue → <UiCard />
  • Use defineProps<T>() to accept data from the parent.
  • Use defineEmits() to send events back to the parent.
  • Use <slot /> when a component should accept flexible inner content.

Layouts — Page Wrappers

A layout is a reusable shell that wraps around your pages. It contains the things that stay constant across multiple pages — like a navigation bar, a sidebar, or a footer — so you don't repeat them everywhere.

What is a Layout?

Without layouts, every page in your app would have to include its own <AppHeader /> and <AppFooter /> at the top and bottom. If you ever changed the header, you'd have to update every single page.

🏠 Analogy for students

Think of a layout as the walls, roof, and foundation of a room. Every room in a hotel has the same structure — four walls, a ceiling, a door. But the content inside each room is different. The "default layout" is the room structure; the page is the furniture inside.

Creating Layouts

Layouts live in the layouts/ folder. The file name becomes the layout name. Nuxt automatically applies layouts/default.vue to every page that doesn't specify a different layout.

layouts/default.vue — the base layout

<template>

  <div class="app-wrapper">


    <!-- This appears on EVERY page that uses this layout -->

    <AppHeader />


    <!-- This is where the page content renders -->

    <main>

      <slot />

    </main>


    <!-- Also on every page -->

    <AppFooter />


  </div>

</template>

🔑
The Key to Layouts: <slot />

The <slot /> inside a layout is where the page content appears. When Nuxt renders a page, it injects the entire page template into the <slot /> of its layout. Everything outside the slot — the header, footer, sidebar — stays fixed.

Different Layouts for Different Pages

Not every page needs the same shell. Your login page probably doesn't need the main navigation. Your admin dashboard might have a sidebar. Nuxt lets you define multiple layouts and assign them per page.

Layout File Contains Used For
layouts/default.vue Header + Footer All public pages (home, about, etc.)
layouts/dashboard.vue Header + Sidebar + Footer Authenticated admin pages
layouts/blank.vue Nothing — just <slot /> Login, register, 404 pages

Applying a Layout to a Page

Use definePageMeta() inside the page's <script setup> to tell Nuxt which layout to use. This is the cleanest, recommended approach.

pages/dashboard/index.vue — specifying a layout

<template>

  <div>

    <h1>Dashboard</h1>

    <p>Welcome back!</p>

  </div>

</template>


<script setup lang="ts">

// This tells Nuxt: "wrap this page in layouts/dashboard.vue"

definePageMeta({

  layout: 'dashboard', // matches the filename: layouts/dashboard.vue

})

</script>


// If you don't specify a layout, Nuxt uses 'default' automatically

⚠️
Important: definePageMeta must be in <script setup>

definePageMeta() is a compile-time macro — it's processed by Nuxt before the app runs. It must be at the top level of <script setup>, not inside a function or conditional block.

Common Student Questions — Layouts

What happens if I don't specify a layout?
Nuxt automatically uses layouts/default.vue. If that file doesn't exist, Nuxt just renders the page without any wrapper.
Can a layout use components?
Absolutely. A layout is just a special component. It can use <AppHeader />, <SidebarNav />, or anything else — they're all auto-imported.
What's the difference between a layout and a component?
Conceptually, a layout is a special type of component that wraps entire pages. The key difference is that pages are assigned to layouts — layouts are not used directly in templates with <TagName /> syntax. Also, layouts use the <slot /> to render the page that's inside them.

Chapter 2 Summary — Layouts

  • Layouts are reusable page shells containing nav, footer, sidebar, etc.
  • They live in the layouts/ folder — filename = layout name.
  • layouts/default.vue is applied automatically to all pages.
  • The <slot /> inside a layout is where the page content renders.
  • Override the layout per-page using definePageMeta({ layout: 'name' }).
  • Use blank.vue for pages that need no shell (login, error pages).

API Services — Talking to the Backend

An API service is a dedicated class responsible for all communication with a backend API. Instead of writing raw fetch calls scattered across your components, you centralize all HTTP requests in one place.

Why Use a Service Layer?

Imagine you're making five different components that all need to fetch customers from your Laravel API. Without a service layer, each component would have its own $fetch call, its own error handling, and its own URL string. If the URL ever changes — or the error handling logic needs updating — you'd have to find and fix all five places.

📞 Analogy for students

Think of the API service as a receptionist at a company. No matter which employee (component) needs to contact an external company (backend), they don't call directly — they go through the receptionist. The receptionist knows the correct number, speaks the right language, and knows what to do when there's no answer (error handling). Everyone benefits from this single point of contact.

Understanding BaseService

In our Nuxt project, all services extend a BaseService class. This class has one method — request() — that handles the mechanics of every single HTTP call. Let's walk through it piece by piece.

① useRuntimeConfig — Where Does the URL Come From?

We never hardcode the API URL in our code. That's because the URL is different depending on the environment — development uses localhost, production uses the real server. useRuntimeConfig() reads the URL from the environment variables.

nuxt.config.ts + .env — environment variable setup

// nuxt.config.ts — tells Nuxt about your env vars

export default defineNuxtConfig({

  runtimeConfig: {

    public: {

      apiBaseURL: '' // default empty — filled by env var

    }

  }

})


// .env file — your actual values (never commit this to git)

// NUXT_PUBLIC_API_BASE_URL=http://localhost:8000/api ← development

// NUXT_PUBLIC_API_BASE_URL=https://api.myapp.com/api ← production


// In BaseService — reading the value:

const runtimeConfig = useRuntimeConfig()

config.baseURL = runtimeConfig.public.apiBaseURL

// Result: baseURL = "http://localhost:8000/api"

🔐
public vs private runtimeConfig

Values under runtimeConfig.public are exposed to the browser. This is safe for your API base URL. If you have a secret key (like a payment gateway secret), put it directly under runtimeConfig (not under public) — it will only be available server-side.

② $fetch — Nuxt's HTTP Client

$fetch is Nuxt's built-in HTTP function. It's a wrapper around the browser's fetch API that works on both the server (SSR) and the client — so your API calls behave consistently regardless of where they run.

It also automatically parses JSON responses, so you get a JavaScript object back without having to call .json() yourself. And you can pass TypeScript generics ($fetch<Customer>) so TypeScript knows the shape of the response.

③ GET vs POST — How Params Are Sent

This is a critical distinction. HTTP has rules about how data is sent, and BaseService respects them:

HTTP Method How params are sent Example URL Result Use Case
GET As query parameters appended to the URL /customers?page=1&search=john Reading / fetching data
POST / PUT As request body (JSON) /customers (data is hidden in body) Creating or updating data
DELETE No body needed — ID is in the URL path /customers/42 Deleting a record
BaseService.ts — the GET vs POST branching logic explained

// BaseService checks the method and places params in the right place:

if (method.toUpperCase() === 'GET') {

  config.params = params // → becomes ?key=value in the URL

} else {

  config.body = params // → becomes JSON body of the request

}


// Example: GET with filter params

this.request('/customers', 'GET', { search: 'john', page: 1 })

// → GET /customers?search=john&page=1


// Example: POST with body data

this.request('/customers', 'POST', { name: 'Jane Doe', email: 'jane@example.com' })

// → POST /customers (with JSON body)

④ Error Handling — The Switch Block

When an API call fails, the server responds with an HTTP status code that tells us what went wrong. BaseService catches every error and converts it into a descriptive JavaScript Error — so the component doesn't have to deal with raw HTTP details.

Here's what each status code means in plain English:

Status Code What it means Who caused it Typical UI response
400 Bad Request The request was malformed or had invalid data Frontend (us) Show a form error message
404 Not Found The requested resource doesn't exist Frontend (us) Show "not found" message or redirect
422 Unprocessable Validation rules failed on the server (Laravel) Frontend (us) Show specific field validation errors
429 Too Many Requests Rate limit hit — too many requests too fast Frontend (us) Show "please wait" notice
500 Server Error Something crashed on the backend Backend (not us) Show "something went wrong, contact admin"
💡
Teaching Tip — 4xx vs 5xx

4xx errors are the client's fault (our frontend sent something wrong or asked for something that doesn't exist). 5xx errors are the server's fault (the backend crashed). This distinction helps developers know where to look when debugging.

Extending BaseService — Creating Feature Services

The BaseService is never used directly by components. Instead, you create a feature service that extends it — like CustomerService. The feature service provides named, readable methods for each API action.

1
Import and extend BaseService

Your new service class extends BaseService, which means it automatically inherits the request() method.

2
Define the TypeScript interface for your data

Describe the shape of the data your API returns. TypeScript uses this to validate your code at compile time.

3
Add a method for each API action

One method per action — getAll(), getById(), create(), update(), remove().

4
Export a singleton instance

export default new CustomerService() — so the whole app shares one instance.

services/CustomerService.ts — a complete feature service

import { BaseService } from './BaseService'


// Step 2: Define the shape of a Customer object

export interface Customer {

  id: number

  name: string

  email: string

}


// Step 1: Extend BaseService

export class CustomerService extends BaseService {


  // Step 3: One method per API action

  getAll() {

    // GET /customers — no extra params needed

    return this.request<Customer[]>('/customers', 'GET')

  }


  getById(id: number) {

    // GET /customers/42

    return this.request<Customer>(`/customers/${id}`, 'GET')

  }


  create(data: Partial<Customer>) {

    // POST /customers — data goes in request body

    return this.request<Customer>('/customers', 'POST', data)

  }


  update(id: number, data: Partial<Customer>) {

    // PUT /customers/42 — data in body

    return this.request<Customer>(`/customers/${id}`, 'PUT', data)

  }


  remove(id: number) {

    // DELETE /customers/42 — no body needed

    return this.request<void>(`/customers/${id}`, 'DELETE')

  }

}


// Step 4: Export one shared instance

export default new CustomerService()

Using a Service in a Component or Page

Once your service is set up, using it inside a page or component is simple and predictable. The pattern is always the same: set a loading state, call the service, handle success, catch errors.

pages/customers/index.vue — fetching data with a service

<script setup lang="ts">

import customerService from '~/services/CustomerService'

import type { Customer } from '~/services/CustomerService'


// Reactive state

const customers = ref<Customer[]>([]) // the list, starts empty

const loading = ref(false) // show a spinner while fetching

const error = ref<string | null>(null) // store error message if it fails


const fetchCustomers = async () => {

  loading.value = true // show spinner

  error.value = null // clear any previous error


  try {

    customers.value = await customerService.getAll()

    // If we reach here, it succeeded ✅

  } catch (e: any) {

    error.value = e.message // the message from BaseService's throw ❌

  } finally {

    loading.value = false // always hide spinner, success or fail

  }

}


// Run when the page/component mounts

onMounted(fetchCustomers)

</script>


<template>

  <div>

    <p v-if="loading">Loading...</p>

    <p v-else-if="error" class="error">{{ error }}</p>

    <ul v-else>

      <li v-for="customer in customers" :key="customer.id">

        {{ customer.name }}

      </li>

    </ul>

  </div>

</template>

✅
The try / catch / finally pattern — always use this

try — run the risky code (the API call).
catch — handle the failure if it throws (store the error message).
finally — this runs regardless of success or failure — perfect for hiding a loading spinner.

Tracing a Full Request — Step by Step

Let's trace exactly what happens when a user opens the customers page and the data loads:

1
Page mounts → onMounted fires

fetchCustomers() is called. loading.value becomes true. A spinner appears in the UI.

2
customerService.getAll() is called

This calls this.request<Customer[]>('/customers', 'GET') inside CustomerService.

3
BaseService builds the config

Gets apiBaseURL from useRuntimeConfig(). Method is GET, so params go into config.params (none in this case).

4
$fetch sends the request

GET https://api.myapp.com/api/customers with Accept: application/json header.

5
Laravel responds with 200 OK

The JSON array of customers comes back. $fetch parses it. request() returns it typed as Customer[].

6
Page receives the data

customers.value is updated. loading.value becomes false. The list renders in the template.

Common Student Questions — Services

Why can't I just use $fetch directly in my component?
You technically can — but if you do it in 10 components, you'll have 10 copies of error handling, 10 different URL strings, and no single place to update them. The service pattern centralizes all of that. It also makes testing much easier — you can mock the service in tests without touching HTTP at all.
What does "extends" mean in class CustomerService extends BaseService?
Inheritance. CustomerService gets all the methods of BaseService for free — including request(). You only write the parts that are specific to customers. This is a core principle of object-oriented programming: don't repeat yourself.
What is Partial<Customer>?
It's a TypeScript utility type that makes every property in Customer optional. This is useful for update requests — you might only be changing the name, not the email. Partial<Customer> allows passing just { name: 'New Name' } without TypeScript complaining about missing fields.
What does export default new CustomerService() mean?
This creates one instance of CustomerService when the file is first imported, and exports that single instance. Every component that imports it gets the same object — this is the Singleton pattern. It saves memory and avoids creating unnecessary duplicate objects.

Chapter 3 Summary — API Services

  • A service centralizes all API calls — components never call $fetch directly.
  • BaseService.request() handles config, baseURL, and error handling for every call.
  • useRuntimeConfig() reads the API base URL from environment variables — never hardcode URLs.
  • GET → params become query string; POST/PUT → params become JSON body.
  • Errors are caught and converted to readable Error objects based on HTTP status codes.
  • Feature services (e.g. CustomerService) extend BaseService and add named methods per resource.
  • Components use try/catch/finally to manage loading, data, and error states.

Teaching Tips for the Classroom

1
Start with Components — they're the building blocks

Have students build a simple BaseCard.vue with a prop and use it on a page. Once they see auto-import working, they'll immediately grasp the power of the Nuxt pattern.

2
Demonstrate Layouts with a live before/after

Show the page without a layout (just raw content), then wrap it in default.vue and show how the header/footer appear instantly. This makes the concept click visually.

3
Walk through BaseService with network tab open

Open the browser DevTools Network tab and call a service method. Show students the actual HTTP request — the URL, the method, the headers — so they can connect the code to the real network call.

4
Simulate errors on purpose

Kill the backend server and have students observe the error catching — the 500 error appearing in the component's error state. Then restore it. Students learn more from seeing errors than from perfect demos.

5
The mental model: one job per layer

Component = what the user sees. Layout = the page shell. Service = talking to the backend. Each layer does exactly one job. If a student's component is doing HTTP calls and also managing form state and also rendering a table — it needs to be split up.
